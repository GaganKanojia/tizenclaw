# rvc-tidl-test

Standalone diagnostic tool that validates the `RVCTizenClawService` TIDL RpcPort interface
exposed by `com.samsung.vr.robot-main` on Samsung robot vacuum cleaners.

Sends `vc_command N` through the TIDL proxy and reports the full roundtrip result with
timing. No internet, SmartThings, or tizenclaw required.

## Regenerating TIDL files with tidlc

All generated files come from `tidl/rvc_local.tidl` using **tidlc 1.10.6** (the version
shipped with the Tizen 7.0 SDK). tidlc is available in Tizen Studio or as a standalone
binary from the SDK tools package.

The current interface definition:

```tidl
interface RVCTizenClawService {
    void ResultCB(string sender, string result) delegate;
    int SendCommand(string rvcCommand, int mode);
}
```

### Proxy files ‚Äî client side (this test tool)

```bash
# Generate proxy files used by rvc-tidl-test (and rvc-local-cli)
tidlc --language c++ --mode proxy \
      --input  tidl/rvc_local.tidl \
      --output tidl/rvc_local

# Produces:
#   tidl/rvc_local.h    ‚Äî RVCTizenClawService proxy class + exception types
#   tidl/rvc_local.cc   ‚Äî Connect(), SendCommand(), parcel serialisation
```

The proxy class connects to the service and calls methods on it. `main.cc` uses it to
send `vc_command N` and wait for the `__Result` parcel.

### Stub files ‚Äî server side (robot-main)

```bash
# Generate stub files that robot-main implements
tidlc --language c++ --mode stub \
      --input  tidl/rvc_local.tidl \
      --output tidl/rvc_local

# Produces:
#   tidl/rvc_local.h    ‚Äî RVCTizenClawService stub base class
#   tidl/rvc_local.cc   ‚Äî RpcPort server registration, parcel deserialisation,
#                         dispatch to OnSendCommand() virtual method
```

The stub generates a `RVCTizenClawService` base class with a pure-virtual
`OnSendCommand(string rvcCommand, int mode)` method. The robot-main process subclasses
it, registers the stub with the Tizen RpcPort framework, and returns an `int` result.
The framework automatically serialises the return value back as the `__Result` parcel.

### Generate both at once

```bash
tidlc --language c++ --mode all \
      --input  tidl/rvc_local.tidl \
      --output tidl/rvc_local
```

> **Important:** Proxy and stub must be generated with the **same tidlc version** (1.10.6).
> A version mismatch changes the method ordinals written into parcels, causing commands to
> be silently ignored or misinterpreted by robot-main.

Key constants in the generated code:

| Symbol | Value | Direction | Meaning |
|--------|-------|-----------|---------|
| `MethodId::__Result` | `0` | server ‚Üí client | Parcel tag for return values |
| `MethodId::__Callback` | `1` | server ‚Üí client | Parcel tag for delegate callbacks |
| `MethodId::SendCommand` | `2` | client ‚Üí server | First int32 in every command parcel |
| `DelegateId::ResultCB` | `1` | server ‚Üí client | ID of the `ResultCB` delegate |
| `TIDL_VERSION` | `"1.10.6"` | both | String written to every parcel header tag |

## Building with GBS

This directory must be a git repository for GBS to accept it.

```bash
# One-time git setup (already done if cloned)
cd /path/to/rvc-tidl-test
git init && git add . && git commit -m "Initial commit"

# Build RPM for aarch64 (Tizen 7.0 device)
gbs build -A aarch64 --include-all

# Build RPM for x86_64 (Tizen 7.0 emulator)
gbs build -A x86_64 --include-all
```

The resulting RPM is at:

```
~/GBS-ROOT/local/repos/tizen7/aarch64/RPMS/rvc-tidl-test-1.0.0-1.aarch64.rpm
```

## Usage

```
rvc-tidl-test [N]     Send vc_command N to robot-main (N = 1‚Äì15, default: 1)
rvc-tidl-test list    Print all command numbers as JSON
```

Step-by-step diagnostics are printed to **stderr**. The final result is one JSON line on
**stdout** ‚Äî suitable for scripting.

### Examples

```bash
# Test vc_command 1 (mapping)
sdb shell "rvc-tidl-test 1"

# Test vc_command 2 (clean-map)
sdb shell "rvc-tidl-test 2"

# Test an unknown command number
sdb shell "rvc-tidl-test 7"

# List all command numbers
sdb shell "rvc-tidl-test list"

# Redirect: capture JSON result only, print diagnostics to terminal
sdb shell "rvc-tidl-test 1 2>/dev/null"
```

### Output format

**Success:**
```json
{"status":"ok","command":1,"action":"mapping","ret":0,"elapsed_ms":142}
```

**Failure:**
```json
{"status":"error","command":1,"message":"timeout ‚Äî no __Result within 5 s"}
```

| Field | Description |
|-------|-------------|
| `status` | `"ok"` or `"error"` |
| `command` | The vc_command number sent |
| `action` | Human-readable name (`mapping`, `clean-map`, or `cmdN`) |
| `ret` | Raw int return value from `SendCommand` (success only) |
| `elapsed_ms` | Total time from connect to result received |
| `message` | Error description (error only) |

### Possible error messages and their meaning

| Message | Cause |
|---------|-------|
| `invalid app ID ‚Äî robot-main not found or not running` | `com.samsung.vr.robot-main` is not installed or not active |
| `permission denied ‚Äî check SMACK label / privileges` | SMACK policy rejects the connection; try running as root |
| `IO error ‚Äî RPC port unavailable` | Low-level socket or send failure |
| `timeout ‚Äî no __Result within 5 s` | robot-main received the command but did not send back a result parcel |
| `invalid protocol response from robot-main` | robot-main sent a malformed or unexpected parcel |
| `not connected ‚Äî Connect() succeeded but port is null` | Internal proxy state error |

## Exit codes

| Code | Meaning |
|------|---------|
| `0` | `SendCommand` completed and `__Result` was received |
| `1` | Any connection, protocol, or timeout error |
